# OpenClaw 飞书授权问题排查与解决方案

## 问题背景

在使用 OpenClaw + 飞书机器人时，AI 读取消息分为两种模式，权限要求不同：

---

## ✅ 被动接收（不需要授权）

以下情况飞书事件会把内容**直接带入 OpenClaw 会话上下文**，LLM 可直接处理，无需额外授权：

- 消息中有明确的**消息引用**（reply）
- 消息中有 **@机器人** 的记录
- 与机器人的**直聊（私聊）**

> 原理：飞书事件推送会把相关消息内容直接带到 OpenClaw，成为会话上下文，LLM 直接处理即可。

---

## ❌ 主动拉取（需要用户 OAuth Token）

以下情况需要调用飞书 API 主动拉取数据，必须以**用户身份**完成 OAuth 授权：

- 未指定明确的消息记录，需要**读取历史消息**
- 需要**搜索聊天记录**
- 其他需要以用户身份发起的 API 请求

> 问题根因：`/feishu auth` 命令只有 **App Owner** 能执行，普通用户无法完成授权，导致卡住。

---

## 解决方案：创建独立飞书机器人应用（App Owner 本人使用）

适用场景：需要主动拉取消息的用户，自己是该飞书应用的 App Owner。

### 步骤

1. **以自己的身份创建飞书机器人应用**
   - 登录飞书开放平台：https://open.feishu.cn/app
   - 创建新的自建应用

2. **配置 OpenClaw**
   - 将新应用的 `appId` / `appSecret` 配置到 OpenClaw
   - OpenClaw 的飞书插件支持配置多个账号

3. **完成用户配对**
   ```bash
   openclaw pairing approve feishu <your_open_id>
   ```
   以自己的飞书 `open_id` 完成配对

4. **完成 OAuth 授权**
   - 以自己为 App Owner 执行 `/feishu auth` 完成授权

5. **配置事件与回调**
   - 进入新建的飞书机器人应用
   - 左侧菜单 → **事件与回调**
   - 订阅方式改为 **「使用长链接接收事件」**（⚠️ 不是 Webhook）
   - 点击验证 → 确认保存

6. **测试与授权**
   - 测试连接（机器人回复即成功）
   - 完成必要的飞书 API 授权（飞书卡片形式推送，按提示操作）

7. **发布新的应用版本**
   - 提交审核并发布，确保权限配置生效

---

## ⚠️ 已知限制：非 App Owner 用户无法完成授权（截至 2026.4.11 未解决）

### 问题描述

当其他用户（非 App Owner）尝试让机器人主动拉取消息时，机器人会提示：

> "你需要先完成授权才能让我以你的身份读取消息哦～发一下「授权」，系统会推送授权卡片给你"

但点击授权后，会报错：

> "❌ This command is restricted to the app owner."

### 根本原因

OpenClaw 当前版本（已验证至 2026.4.11）的设计限制：

- `/feishu auth` 触发的 OAuth 授权流程**仅限 App Owner** 执行
- 普通用户（即使在应用可用范围内）无法自助完成授权
- OpenClaw **没有提供 "fallback to owner token" 的配置项**，无法让其他用户借用 Owner 的 token 来拉取消息
- `openclaw pairing approve` 需要对方先发起配对请求才能执行，无法强制绕过

### 尝试过但无效的方案

| 方案 | 结果 |
|---|---|
| 修改飞书应用可用范围为「所有员工」并重新发布 | 无效，仍报 Owner 限制 |
| 撤销授权后重新授权 | 无效 |
| 升级 OpenClaw 到最新版本（2026.4.11） | 无效，问题依然存在 |
| openclaw pairing approve + open_id | 报错：No pending pairing request found |

### 当前可行的变通方案

**方案一：非 Owner 用户改用「被动接收」方式（推荐）**

不要发"读取聊天记录"这类主动拉取请求，改为：
- **引用具体消息** + @ 机器人，让机器人处理被引用的内容
- 把需要整理的内容**直接复制粘贴**给机器人，由机器人处理

**方案二：非 Owner 用户创建自己的飞书应用**

参考上方「解决方案」章节，非 Owner 用户自己创建独立飞书应用并绑定到 OpenClaw，自己成为 App Owner 后即可完成授权。

> ⚠️ 此方案需要额外的飞书应用管理和 OpenClaw 配置成本，适合对主动拉取消息有强需求的用户。

## 补充：飞书应用「智秀」主动拉取失败的根本原因

> 适用场景：模糊搜索消息、读取群聊天记录等主动拉取操作

### 根本原因

**原因一：「应用身份权限」缺失**

飞书应用需要开通以下两个**免审权限**才能读取用户基本信息及通讯录：

| 权限名称 | 权限标识 | 访问凭证 | 说明 |
|---|---|---|---|
| 获取用户基本信息 | `contact:user.base:readonly` | user_access_token | 返回用户名、英文名、头像 |
| 获取通讯录基本信息 | `contact:contact.base:readonly` | user_access_token | 关联前者，无法单独移除；可查询用户、部门、通讯录数据范围变更等 |

> 两项权限均为**免审权限**，风险等级安全，直接开通即可，无需审核。

**原因二：「应用身份权限」可访问数据范围错误**

在飞书开发者后台 → 权限管理 → 通讯录 → **「权限可访问的数据范围」** 需设置为：

- ✅ **与应用的可用范围一致**（推荐）
- ❌ 不要设置为「部分成员」（会导致部分用户无法读取）

### 解决方式

重新创建飞书应用并绑定 OpenClaw 后，正确配置上述两项权限及数据范围，问题已解决。

## 补充：复用旧飞书应用导致机器人无响应

### 问题描述

复用了此前通过「**飞书秒搭**」创建并已绑定过 OpenClaw 的飞书应用「Slots的助手」，导致机器人**经常无响应**。

### 根本原因

飞书事件订阅方式的特性问题：

- 「长链接接收事件」模式下，飞书应用的机器人事件会**绑定到建立连接的 OpenClaw 实例**
- 复用旧应用时，事件可能仍指向**旧的 OpenClaw 实例**，而非当前运行的实例
- 导致消息发出后，旧实例尝试处理但无法响应，当前实例收不到事件

### 解决方式

> ⚠️ **建议：不要复用已绑定过其他 OpenClaw 实例的飞书应用**

每次为新的 OpenClaw 实例创建**全新的飞书应用**，避免事件路由混乱。

### 安全备注

> 🔒 **请通知飞书管理员删除已废弃的飞书应用**，避免遗留应用的权限和 Token 造成潜在安全风险。

---

## 关键区别总结

| 场景 | 是否需要授权 | 可用人员 | 原理 |
|---|---|---|---|
| 回复引用的消息 | ❌ 不需要 | 所有人 | 飞书事件直接带入上下文 |
| @机器人发消息 | ❌ 不需要 | 所有人 | 飞书事件直接带入上下文 |
| 与机器人私聊 | ❌ 不需要 | 所有人 | 飞书事件直接带入上下文 |
| 读取群历史消息 | ✅ 需要 | **仅 App Owner** | 调用飞书 API，需用户 OAuth Token |
| 搜索聊天记录 | ✅ 需要 | **仅 App Owner** | 调用飞书 API，需用户 OAuth Token |
