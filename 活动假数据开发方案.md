# 活动前端独立开发方案

## 假数据驱动 · 前后端并行

<br>

<span class="tag">WorldTourCasino</span>
<span class="tag">前端架构</span>
<span class="tag">开发效率</span>

---

## 背景：为什么需要假数据？

<br>

```
传统流程                          并行流程
─────────────────                 ─────────────────────────────
后端开发协议      ──→              后端：定义协议字段
      ↓                           前端：同步构建假数据
前端等待接口                             ↓
      ↓                           前端独立自测完成
联调 → 修 bug → 上线               后端就绪 → 联调 → 上线
─────────────────                 ─────────────────────────────
  串行，周期长                       并行，效率翻倍
```

<br>

> 核心思路：**让 Request 层可插拔**，假数据与真实请求共用同一套处理逻辑，联调时一行开关切换，零改造成本。

---

## 活动代码架构总览

<br>

| 层次 | 文件 | 职责 |
|---|---|---|
| **活动主体** | `XxxActivity.js` | 生命周期、支付验证、资源下载 |
| **数据层** | `XxxActivityData.js` | 封装服务器数据，对外提供 getter |
| **请求层** | `XxxActivityRequest.js` | 所有接口调用入口，**假数据注入点** |
| **枚举层** | `XxxActivityEnums.js` | ClaimType、EventType、错误码 |
| **UI 管理** | `XxxActivityUIMng.js` | 弹板管理，解耦业务与 UI |
| **控制器** | `controller/` | 各界面渲染与交互 |

---

## 数据流向

<br>

```
                    FAKE_MODE = true
                         │
服务器响应               ▼
    │          FakeDataMng._buildResult()
    │               │
    └───── Request.handler ──────┐
                                 │
                           syncData()
                                 │
                         ActivityData (getter)
                                 │
                        Controller → 渲染 UI
```

<br>

> `handler` 回调只写一次，真实与假数据完全共用，联调时删除 `FAKE_MODE` 分支即完成切换。

---

## 新建活动：五步到位

<br>

**① 复制模版**
```bash
cp -r scripts/activity_js_template/ src/activity/xxx_activity/
```

**② 全局重命名**　`Template` → `Xxx`（保持大小写规律）

**③ 填写 Enums**　ClaimType、EventType、错误码

**④ 实现 ActivityData**　`setActivityData` 初始化 + `syncData` 增量更新 + 各 getter

**⑤ 新建 FakeDataMng**　填入假数据，开始独立开发

## Request 层：标准假数据接入写法

```javascript
// TODO: 上线前删除以下两行
var XxxFakeDataMng = require("./XxxActivityFakeDataMng");
var FAKE_MODE = true;

XxxActivityRequest.claimFinalReward = function (activity, callback, errorCallback) {
    var sendData = { claimType: XxxClaimType.CLAIM_FINAL_REWARD };

    // handler 只写一次，真实/假数据共用
    var handler = function (errorCode, resultData) {
        game.popupMan.closeIndicator();
        if (errorCode) { cc.isFunction(errorCallback) && errorCallback(errorCode); return; }
        activity.getActivityData().syncData(resultData);
        cc.isFunction(callback) && callback(resultData);
    };

    game.popupMan.popupIndicator("", 0, true);
    if (FAKE_MODE) { XxxFakeDataMng.claimSpecialActivityFake(sendData, handler); return; }
    game.aUtil.claimSpecialActivity(activity, sendData, handler);
};
```

## FakeDataMng：三大设计亮点

<br>

### 1 · 延迟模拟

```javascript
var FAKE_DELAY_MS = 300; // 模拟真实网络延迟，使 indicator 正常显示
```

> 无延迟时 indicator 瞬间消失，可能隐藏时序 bug，建议 ≥ 300ms

<br>

### 2 · 状态轮转

```javascript
// 每次点击自动返回下一条数据，模拟多次操作后的不同状态
entry.index = (entry.index + 1) % entry.fakeData.length;
```

## 假数据覆盖清单（以 CasinoRoyale 为例）

| claimType | 需模拟的状态 |
|---|---|
| `CLAIM_BUILD` | 建造成功 → 道具减少、buildIds 更新 |
| `CLAIM_AUTO_BUILD` | 批量建造后道具剩余量 |
| `CLAIM_FINAL_REWARD` | 首次领取 / 已领取（needClaimFinalReward 变化） |
| `CLAIM_UPDATE` | 普通刷新 / 小游戏 leftTime 到期可领取 |
| `CLAIM_BONUS` | 挂机收益 / 清扫 / 送餐 三种 gameType |
| `CLAIM_GAME` | 免费进入 / 购买后进入 / 小游戏结算 |

## ActivityData：标准 syncData 实现

通用增量同步，**模版已内置**，各活动直接继承，无需重复开发：

```javascript
syncData: function (resultData) {
    var keys = Object.keys(resultData);
    for (var i = 0; i < keys.length; i++) {
        var key = keys[i];
        if (cc.isArray(resultData[key])) {
            this.activityData[key] = [];
            game.util.deepCopyObject(this.activityData[key], resultData[key]);
        } else if (cc.isObject(resultData[key])) {
            this.activityData[key] = {};
            game.util.deepCopyObject(this.activityData[key], resultData[key]);
        } else {
            this.activityData[key] = resultData[key];
        }
    }
}
```

> 支持数组、对象、基础类型的增量覆盖，`nextSectionId` 等特殊字段在子类中扩展处理

## 开发流程总览

```
需求确认
 ├─ 后端：定义协议字段，输出接口文档
 └─ 前端：根据协议构建 FakeDataMng
          ↓
  独立开发（FAKE_MODE = true）
  ├─ 覆盖正常流程
  ├─ 覆盖边界状态（道具不足 / 已领取 / 已结束）
  └─ 通过 fakeErrorOverride 覆盖错误码路径
          ↓
  后端就绪 → 联调（FAKE_MODE = false）
          ↓
  上线前清理
  └─ 删除 FakeDataMng 文件
  └─ 删除 require 行 + FAKE_MODE 变量
```

# 总结

<br>

| | 价值 |
|---|---|
| **并行开发** | 前端不等后端，开发周期压缩 |
| **零联调成本** | 一行开关切换，handler 共用 |
| **全路径覆盖** | 正常 + 边界 + 错误码均可模拟 |
| **上线零残留** | 模板约定删除规范，不污染生产代码 |
