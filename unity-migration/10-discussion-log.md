## 一、已定案决策

### DEC-01 过渡期发布形态：纯迁移、不混跑、终点一次性平替 ✅

**定案时间**：2026-07-08

- 迁移过渡期内，新关卡**仍基于 Cocos 2022 轨开发**，随后陆续迁移；
- 不做 Unity/Cocos 混合运行（不做 UAAL 嵌入、不做双端并行开发）；
- 最终迁移完成后**择机直接平替发布**，之后新关卡研发彻底切换到 Unity。

**否决的备选**（留档）：
- ❌ 双端各做一份新关卡：成本翻倍，且迫使 Unity 框架贴近 Cocos 语义，干净层失去意义；
- ❌ Cocos 冻结新关等 Unity 壳整体接管：被大厅/活动/支付等范围外依赖卡死；
- ❌ UAAL 混合运行：Web 端（Facebook Canvas）Unity WebGL 与 Cocos-html5 共存不可行，桥接层是额外工程量。

**连锁推论**：
1. 原方案 L1→L2→试点→批量的串行 Phase 结构在此模式下成立，路线图只需追加"追赶收敛"尾段；
2. **迁移目标是移动的**：终点 = 95 个存量 + 迁移期新增产量（95 + r·T），流水线吞吐必须显著大于新关产率 r 才能收敛；
3. 平替 = 大爆炸发布，无灰度混跑期，**等价性验证的自动化覆盖率决定成败**。

### DEC-02 迁移路线：以"规格提取 + AI 重实现"为主干，代码机翻降级为逃生舱 ✅

**定案时间**：2026-07-08

**背景**：用户明确"迁移方式不限于代码级翻译，可借助 AI 将已有关卡用 Unity 重新实现"。真正的不变量只有两个：存量关卡行为在 Unity 上重现 + 新关卡在 Unity 上高效研发。

**三条路线对比**：

| | 路线 A：代码机翻（原方案） | 路线 B：AI 规格重实现 | 路线 C：混合（提取+重实现） |
|---|---|---|---|
| 搬运物 | 代码结构（JS→C# 逐行对应） | 行为规格（关卡"是什么"） | 确定性部分机器提取，变异部分 AI 重写 |
| 固定成本 | 极高（L1 全量 ~400 API + 映射表） | 中（干净 L2 + 规格模板 + 回放 harness） | 同 B + 规格提取工具 |
| 单关边际成本 | 低且确定 | 中且有方差 | 介于两者之间 |
| 产物质量 | 9 项实现负债原样进 Unity | 干净层代码，与新关卡同方言 | 干净层代码 |
| 保真度保证 | 构造性（逐行对应） | 验证性（回放比对 + 视觉验收） | 验证性 |

**选择路线 C 的依据**（本代码库对规格重实现异常友好，均有调研数据支撑）：
1. **规格大半已是结构化数据**：CPA 架构下一个关卡 = machineConfig（100+ 字段）+ FlowDispatcher 流程树声明（DSL 数组）+ 组件注册表 + Action 列表，全部声明式，可机器解析提取，不需要 AI 读懂 4000 行代码；
2. **代码价值密度低**：40-50% 样板（干净 L2 直接吸收）；同名 Process 复制粘贴变异率 47-61%，关卡个性只在变异点上——逐行翻译等于花高价搬运复制粘贴；
3. **逻辑等价有天然裁判**：协议不变，录制 S2CSpin 序列双端回放比对，逻辑保真不靠"相信 AI"；
4. **领域知识已有文档**：FreeSpin 12 步节奏模型、win_effect 4 级分级等注释/文档是 AI 重实现的关键输入，且已存在。

**路线 C 的具体形态**：
1. 干净 L2 先行（设计不再受 compat 制约）；
2. 规格提取器：解析 machineConfig / 流程树 / 组件表 / Action 表 → 关卡规格文档 + 半成品 C# 骨架 + 待填空演出点清单；
3. AI 重实现：规格 + 原演出代码 + 领域文档 → 在干净 L2 上重写变异部分；
4. 双重验收：回放 harness 裁判逻辑等价 + 双端录屏对比裁判演出等价；
5. 机翻保留为逃生舱：个别极端复杂关卡（如 GlitzyHits 4253 行级别）允许局部代码级翻译兜底，**L1 只按兜底所需最小子集建设，不再是 400 API 全量**。

**已识别的弱点**（不回避）：
- 演出手感：变异率最高的部分是演出代码（帧级 delay、缓动、粒子时机），无结构化规格、回放裁判不了，只能双端录屏对比 + 人工验收，是压不掉的人工底座；
- 质量方差：AI 重写有长尾，个别关卡可能返工 2-3 轮，吞吐估算按 P75 而非均值；
- 提取死角：关卡内的协议 hack、直改老框架数据（729 处双向耦合的一部分）提取器可能漏掉 → 回放 harness 必须先行，作为安全网。

**决定性实验（待执行）**：选 292（最小，9 文件）+ 一个中等关卡，同一关卡 A/C 两条路线各走一遍，量三个数：单关工时、回放比对通过率、演出验收返工轮数。数据落地后取舍不再是观点问题。

### DEC-03 对现有 unity-migration 方案（v1.0）的继承/重谈判定 ✅

**继承（不重做）**：
- 调研数据层：~400 API、11.4 万次调用统计、Top 20 覆盖 65% 的集中度分析（01 章 1.5 节）；
- 范围决策：只迁 95 个 2022 轨关卡、140 个遗留关卡不碰、协议层不动只换展示层（D7）；
- L2 既定决策：三阶段初始化保留、SlotAction 协议保留、Reel 从零构建（D4）、DOTween 替代 cc.Action（D2）；
- 试点→批量的验证方法论（08 章）。

**重谈（结构性弱点）**：
1. "迁移期间新关卡持续产出"的追赶问题完全缺席 → 新增 G4；
2. L1 兼容层地位：随 DEC-02 从"Phase 1 前置的生产主干"缩回"逃生舱最小子集"；
3. L2 深度不足：流程引擎仍是 callback 直译思路，配置资产化 / async 化 / 时间轴化未体现 → E 档条目逐条重设计。

## 二、讨论中条目

### N1 规格提取器设计 🔄（第 3-4 轮，已修正为"静态+动态"双源提取）

**解剖样本**：292 Rose Romance（最小模块化关卡，1142 行：Config 330 + Scene 812）。

**关卡代码五层解剖**（提取策略随层不同）：

| 层 | 内容 | 占比 | 提取方式 |
|---|------|------|---------|
| ① 配置层 | machineConfig ~180 字段赋值 | ~29% | 纯机械（AST） |
| ② 装配层 | 组件注册表 / 流程树数组 / SA.new 注册 | ~10% | 纯机械（强公约模式） |
| ③ 数据逻辑层 | 改写 receivedSpinResult、影响流程的返回值 | ~15% | 机械识别 + AI 翻译，回放可裁判 |
| ④ 演出编排层 | 演出动作调用（createFlow 链 / 回调嵌套 / runAction 链，不限风格） | ~40% | **动态录制为主 + AI 重实现**（见下） |
| ⑤ 常量映射层 | COLLECT_TYPE_MAP 等 | ~5% | 纯机械 |

**用户三次关键质疑及修正（第 4-5 轮）**：

1. ❌ 纯静态提取只有代码结构，业务意图零体现，AI 靠读代码考古无法可靠还原 → **修正**：不追求从代码反推意图——可运行的 Cocos 版本本身就是可执行的需求说明书，新增动态提取通道把运行时行为变成 AI 可消费输入；
2. ❌ createFlow 并非标准编排，大量关卡是回调金字塔，静态分析无法从中恢复时间轴 → **修正**：时间轴不再从代码提取，从运行时录制，对代码风格无感；
3. ❌ 录屏不合理：上下文太重（视频是 AI 最昂贵最低密度的输入）、存储管理负担大、盖不住（像素丢失节点层级/zOrder/被遮挡的状态变化）→ **修正（用户方案，优于原 monkey-patch 设计）**：静态分析先行，提取器**基于静态分析结果给关卡代码注入定制化运行时记录代码**，用结构化日志录制代替视频录制，日志可分片、按需、渐进式录制。同时修复了 monkey-patch 的内伤（只 patch util 汇聚函数，不经过它们的行为全是盲区）。

**定型架构：静态分析 → 插桩计划 → 定制化注入 → 分片日志录制**

三级探针（粒度递进）：

| 层级 | 插桩点 | 由谁决定 | 何时启用 |
|------|--------|---------|---------|
| P0 基础 | util 汇聚函数（playAnimEasy/playEffect/setNodeVisible/delayCall/runAction…） | 全局固定 | 始终 |
| P1 结构 | 静态分析逐关卡定制：③④层方法入口/出口、**if/else 分支点（含条件求值结果——补上"为什么"）**、**匿名回调实际触发时刻（回调金字塔时序自己浮现）**、receivedSpinResult/this._xxx 状态写入 | 提取器按 AST 生成 | 默认开 |
| P2 稠密 | 指定方法行级 trace（语句执行序+关键变量值） | 按需点名 | 重实现遇歧义时补录 |

- 实现：acorn AST transform 注入探针 → 录制版构建（instrumented bundle，构建变体不进仓库）→ 跑场景注入 harness；探针写 JSONL（帧号+时间戳双标记），按场景分片；
- 渐进工作流：默认 P0+P1 → AI 重实现遇歧义 → 点名方法开 P2 重录相关场景 → 补高密度切片。**录制不足从设计风险降级为运营成本（触发补录即可）**；上下文经济：AI 每次只取当前工作切片；
- **日志成为全流水线通用语言**：提取阶段=行为规格；重实现阶段=AI 参考资料；验收阶段=双端 diff 基质（Unity 侧经同语义 logging facade 发同名事件，JSONL 直接 diff——N2 的比对基质就此确定）；Unity 干净层可将同套语义事件做成常驻调试设施；
- 人工视觉验收保留为验收阶段活动（QA 双端并排试玩），不再是提取阶段产物；可选：日志事件触发的关键帧截图（待定）。

**规格包 schema（第 5 轮修订）**：`spec_<id>/` = manifest + 6 个结构化 JSON + choreography/（.src.js + .intent.md）+ scenarios/（脚本化 spin 结果）+ probes.json（本关插桩计划）+ traces/（分场景分片 JSONL 日志）。~~recordings/ 录屏~~ 已移除。

**已识别死角/边界**：条件配置（isHighResolution 等保留原表达式+环境标记）；方法体内动态 require；Action↔Scene 数据脐带（记录依赖边）；轨迹不含"为什么"（靠场景标签关联）；场景枚举组合爆炸（单变量覆盖+高频组合，遗漏分支进 manifest 风险清单）。

**场景覆盖问题的解法（第 6 轮，回应用户质疑④"特定条件组合触发的表现录不全"）**：

核心区分：静态分析不能"覆盖"所有情况（无法替你执行到分支），但能"枚举"所有情况（P1 分支插桩点集合 = 分支全集 = 完整分母）——把"录不到"从暗伤变成明账。

**分支清单闭环**：静态枚举分支（分母）→ 场景合成（分子）→ 插桩运行测覆盖 → 覆盖率报告 → 未覆盖分支逐条处置。

有利的结构性事实：赌场游戏客户端不产生随机性，玩法触发全是服务器数据的函数，且注入点在 ClassicSlotMan 可伪造任意数据 → 触发条件可从谓词**反向求解**（实测谓词多为平凡字段比较，字段级反解+枚举即可，无需通用约束求解器）。

三类难录情况的处置：
1. **跨局累积状态触发** → 场景序列（连续注入多个 spin 走完累积）+ **状态短路**（伪造 roomEnterInfo 让关卡"出生即攒满"，进房数据注入为 harness 标配）；
2. **反解不动的复杂谓词** → **生产流量回放**：协议不变，线上真实玩家 S2CSpin 序列可直接捞取回放，玩家群体替我们穷举稀有组合；
3. **残余未覆盖分支** → 降级为"代码即规格的保守直译"（按 ③ 层处理）+ 未验证标记进人工 review 队列 + **Unity 版保留同款 P1 分支探针，上线后首次命中即遥测上报**（验证期延伸到发布后受监控运行）。

诚实边界：目标是分支覆盖非路径覆盖（组合指数爆炸），组合效应差异靠 N2 双端 diff / 人工验收 / 上线遥测三道后卫兜底，取舍明写进 manifest。

方法论出处：分母/分子即 istanbul/JaCoCo 类覆盖率工具原理；生产流量回放是协议不变场景的 traffic replay 标准打法——均为成熟技术的嫁接，红利在于客户端是纯确定性数据消费者。

**遗留待拍板**（不阻塞，可后置到 spike 后用实物定）：
1. intent.md 生成时机（提取阶段批量预生成沉淀玩法模式库 vs 重实现阶段现场生成）；
2. 提取完整性验收标准：AST 遍历零"漏网行"（每行必须被某 bucket 认领，漏网即报错）；
3. 覆盖率及格线：分支覆盖率达到多少才允许进入重实现（含生产流量回放补录后）；
4. 是否要日志事件触发的关键帧截图作轻量视觉锚点。

## 三、讨论日志

### 2026-07-08 第 1 轮：范围切换 + 发布形态

### 2026-07-09 第 8 轮：工具链立项 —— Schema-first 决策 + 双样本适配性调研

- 用户确认：新工具放 `tools/analyzer`，先做完 analyzer 再推进；核心疑问——独立做 analyzer 会不会与后续规格包脱节？
- **定案：Schema-first**。规格包 schema（spec-package.schema.json，版本化）是第一个交付物，先于工具代码存在；analyzer 不是独立工具而是规格包的第一个 producer；`spec_<id>/` 是 analyzer/recorder/coverage-verifier 三工具的共享工作产品（analyzer 写静态槽位+probes.json，recorder 追加 traces/，verifier 读包出报告）；槽位可空不可缺（schema 完备性与实现进度解耦）；schema 演进走版本号。
- **双样本解剖调研**（2 agent 并行细读，DashingWin 单文件 1373 行 + 287 模块化 16 文件 2076 行），三个结构性缺口：
  1. **单文件关卡 60% 代码在原 schema 无处安放**：module.exports 后 6 个内联类承载主体行为；零 SlotAction 公约；mixed（数据+演出缠绕）占 38.6%；"overrides=scene 方法"假设崩塌 → 类成为一等公民，overrides 扩展为 {className, baseClass, methodName, code}；
  2. **五层模型漏了第⑥层 UI 数据绑定**：controller（读全局数据→写 label）③/④ 两头不靠，双样本同时命中 → 新增 ui-bindings 槽；
  3. **三份契约比预想宽**：server-contract 有逃逸入口（SlotMan/SlotConfigMan 全局读，需 global-reads）；④层 API 白名单过窄（setLocalZOrder/runAction/NumberAnimation/delayCall 等 15 类会被误判 mixed）；资源名动态拼接只能记模式。
- 零散关键发现：派生状态契约（287 scene 产出 11 字段被 4 组件经 this.context.context 消费——SlotAction"状态外泄"问题的关卡级版本，需 state-contract.json）；伪常量（模块级可变状态被运行时改写）；Math.random 非确定演出点（nondeterminism.json，回放容差输入）；外部 editorConfig 双写契约（external-config.json）；copied-super-body 模式（整段复制基类再微调，规格宜存 diff）；branches 需计入 ||回退隐式分支与位运算标志展开。
- **Schema v0 定型**：manifest / config / assembly / **classes（一等公民，每方法带 data|choreo|mixed|ui-binding 层标注+状态写入点）** / **state-contract** / constants / resources / server-contract / **external-config** / branches / **nondeterminism** / probes / traces（加粗为本轮新增槽位；原 overrides/choreography/ui-bindings 独立槽合并进 classes 的方法层标注）。
- Schema 设计铁律：槽位跟着代码真实形态走而非框架理想公约走——公约只是 clean 路径，每种越轨模式都要有显式收容槽，否则覆盖率报告系统性说谎。
- analyzer 第一个提交物：正式 JSON Schema + 人工对照规格（292 精简版先行，DashingWin 难点版后补），然后才写解析代码。
